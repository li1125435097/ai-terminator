实现一个能够自主感知、决策、执行并复盘的测试与开发Agent，核心在于构建“Agent（智能体）+ Skills（技能/工具能力）”的架构。针对你的需求，可以将其拆解为以下落地步骤：

### 一、 核心架构与技能池设计
Agent 的核心运作公式为：**测试场景 = 1个Agent + N个Skills + 任务流**。Agent 负责拆解任务（Plan）、调用 Skills（Act）并判断结果是否达标（Check）。你需要先沉淀以下原子能力（Skills）：
1. **禅道交互 Skill**：调用禅道 API 或数据库，查询指派给当前用户的 Bug 列表，获取 Bug 详情及复现步骤。
2. **Cursor 调用 Skill**：通过 CLI 命令或 API 接口，将 Bug 描述、复现步骤及代码上下文传递给 Cursor，触发其自动生成或修复代码。
3. **后端测试 Skill**：包含数据库操作（造测试数据、清理数据）和单元测试框架（如 pytest/unittest）的调用能力，用于验证接口逻辑。
4. **前端测试 Skill**：集成浏览器自动化测试工具（如 Playwright、Selenium），模拟真实用户操作进行页面校验。

### 二、 分模块落地执行策略

#### 1. 禅道 Bug 监控与获取
通过 MCP 协议或 API 打通与禅道的连接。Agent 每天定时触发，查询禅道系统中的 Bug 数据。获取到 Bug 后，Agent 需进行初步的文本分类和去重，提取出核心复现步骤和预期结果。

#### 2. 后端：单元测试与数据构造
当 Agent 识别出后端相关 Bug 并调用 Cursor 生成代码后，需要自动验证修复效果：
*   **造数与清理**：利用数据构造 Skill，通过 MySQL/Redis 客户端或 Faker 库自动生成测试所需的账号、订单等数据，并在测试结束后自动清理，确保环境纯净。
*   **接口自动化测试**：调用 pytest 或 unittest 等断言校验工具，对修复后的接口发起请求，校验状态码、业务字段及数据库数据的一致性。

#### 3. 前端：浏览器自动化测试
针对前端 UI 的 Bug，Agent 可调用 Playwright 等执行引擎。它可以模拟真实用户在浏览器中的操作（如点击、输入、页面跳转），捕获页面截图、元素识别结果及 JS 错误日志。若遇到复杂的视觉问题，还可以借助 MCP 服务器（如 ui-ticket-mcp）获取元素的 DOM 结构、CSS 计算样式及边界框等精确元数据，辅助 AI 定位并修复问题。

### 三、 推荐落地路线图
为了避免一步到位带来的复杂性，建议按照以下阶段逐步推进：
*   **阶段一（环境适配与单技能打通）**：配置大模型 API 及 MCP 协议通信链路，先实现“禅道查 Bug”和“后端造数跑单元测试”这两个独立技能的稳定运行。
*   **阶段二（Agent 规则定制与串联）**：定制 Agent 的任务拆解与异常处理规则。实现“查 Bug → 调 Cursor 改代码 → 自动跑后端单元测试”的完整单链路闭环。
*   **阶段三（前端接入与多 Agent 编排）**：引入前端浏览器自动化测试 Skill，并将流程扩展为多 Agent 协作（如：造数 Agent → 修复 Agent → 后端测试 Agent → 前端测试 Agent），最终接入统一的调度平台，实现一键式自动化执行与报告生成。

### 1. 禅道对接（感知层）
禅道是 Agent 的触发器，推荐优先使用官方 API，如果版本较老再考虑数据库直连。
*   **首选：禅道 OpenAPI (v2)**
    *   **核心接口**：`GET /api.php/v2/bugs`（支持按指派人、状态、创建时间等过滤）。
    *   **鉴权方式**：推荐使用 `Token` 鉴权（在禅道后台生成 Access Token），比传统的 Session/Cookie 更适合自动化脚本。
    *   **优势**：官方支持，字段规范，不会因禅道版本升级导致解析失败。
*   **备选：数据库直连 (MySQL)**
    *   **核心表**：`zt_bug`（Bug主表）、`zt_history`（Bug修改历史）。
    *   **适用场景**：禅道版本过老不支持 API，或需要极高频的复杂联合查询。
    *   **注意**：务必使用**只读账号**连接，避免 Agent 误操作污染禅道数据。

### 2. Cursor 调用（执行层）
Cursor 本身没有官方的“AI 编程 API”，我们需要通过其命令行工具（CLI）来驱动它。
*   **核心工具**：Cursor CLI (`cursor-agent` 或 `cursor --agent`)
*   **调用方式**：通过 Python/Node.js 的 `subprocess` 或 `child_process` 执行 Shell 命令。
*   **关键参数**：
    *   `--prompt`：传入格式化好的 Bug 描述和复现步骤。
    *   `--model`：指定使用的模型（如 `claude-3.5-sonnet` 或 `gpt-4o`）。
    *   `--cwd`：指定当前项目的根目录，确保 Cursor 能读取到正确的代码上下文。
*   **落地技巧**：在调用前，先让 Agent 用 `git diff` 或 `git status` 获取当前未提交的代码变动，作为上下文一并喂给 Cursor，防止它改错文件。

### 3. 后端测试与数据构造（验证层）
后端测试的核心是“环境隔离”和“数据自洁”。
*   **测试框架**：**Pytest** (Python) 或 **JUnit 5** (Java)。
    *   *推荐 Pytest*：生态极其丰富，且非常适合 Agent 用自然语言生成测试用例。
*   **接口测试库**：**Requests** (Python) 或 **RestAssured** (Java)。
*   **数据构造与清理**：
    *   **造数**：**SQLAlchemy** (ORM 方式造数，更安全) 或 **Faker** (生成假数据)。
    *   **事务回滚**：在测试框架的 `fixture` 或 `setup/teardown` 中开启数据库事务，测试结束后**自动 Rollback**，确保不会在测试库里留下脏数据。
    *   **容器化（进阶）**：如果条件允许，使用 **Testcontainers** 为每次测试动态拉起一个临时的 MySQL/Redis 容器，测试完直接销毁，实现绝对的环境隔离。

### 4. 前端自动化测试（验证层）
前端测试需要模拟真实浏览器环境，并让 Agent 能“看懂”页面。
*   **核心引擎**：**Playwright** (强烈推荐)。
    *   *为什么选它*：对异步加载、Iframe 支持极好，且自带强大的**自动等待（Auto-wait）**机制，能大幅减少 Agent 写出的测试脚本因为超时或元素未加载而报错的概率。
*   **辅助工具**：
    *   **Playwright Codegen**：Agent 在生成测试代码时，可以先调用 Codegen 录制基础操作，再让大模型进行优化和断言补充。
    *   **视觉回归测试**：集成 `playwright-screenshot`，在测试失败时自动截取全屏或元素截图，作为 Bug 修复失败的证据反馈给 Agent。

### 5. Agent 核心编排框架（大脑层）
你需要一个框架来把上面这些零散的工具串联起来，实现“计划-执行-检查”的闭环。
*   **推荐框架**：**LangGraph** (基于 LangChain) 或 **CrewAI**。
    *   *为什么选 LangGraph*：它支持**状态机（StateGraph）**。你的工作流（查禅道 -> 调 Cursor -> 跑后端测试 -> 跑前端测试）是一个有明确先后顺序和条件判断（比如后端测试失败就不跑前端了）的流程，LangGraph 的节点和边非常适合这种严谨的工程化落地。
*   **通信协议**：**MCP (Model Context Protocol)**。
    *   建议将“禅道查询”、“数据库操作”、“Playwright 执行”封装成独立的 MCP Server。这样你的 Agent 大脑可以专注于决策，而具体的工具调用通过标准 MCP 协议对接，后续如果要换测试工具，只需替换 MCP Server，无需重写 Agent 核心逻辑。
